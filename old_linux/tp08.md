Rapport — TP Réseaux: NAT, Pare-feu et mini DMZ

1) Topologie et adressage

Schéma logique
lan_int (10.10.1.0/24) ↔ fw ↔ dmz (10.10.2.0/24)

- fw (routeur/pare-feu)
  - LAN (lan_int) = 10.10.1.1/24 (eth0)
  - DMZ (dmz) = 10.10.2.1/24 (eth1)
- lan-cli (client LAN)
  - 10.10.1.10/24, gateway 10.10.1.1
- dmz-srv (serveur DMZ)
  - 10.10.2.10/24, gateway 10.10.2.1

VirtualBox (VMS éteintes)
- fw:
  - Adapter 1: Internal Network, Name lan_int
  - Adapter 2: Internal Network, Name dmz
- lan-cli:
  - Adapter 1: Internal Network, Name lan_int
- dmz-srv:
  - Adapter 1: Internal Network, Name dmz

2) Étape 1 — Configuration IP de base

Objectif
Attribuer des IP statiques et vérifier la connectivité locale avec fw.

Commandes et fichiers

Identifier les interfaces (sur chaque VM):
```bash
ip -br addr
ip -br link
```

lan-cli (/etc/network/interfaces):
```text
auto lo
iface lo inet loopback

auto eth0
iface eth0 inet static
    address 10.10.1.10/24
    gateway 10.10.1.1
```
Appliquer:
```bash
sudo rc-service networking restart
sudo rc-update add networking boot
```

fw (/etc/network/interfaces):
```text
auto lo
iface lo inet loopback

    address 10.10.2.1/24
```


3) Étape 2 — Routage simple entre LAN et DMZ

Objectif
Permettre lan-cli ↔ dmz-srv via fw.

Commandes

Activer le routage IPv4 sur fw:
```bash
# temporaire
sysctl -w net.ipv4.ip_forward=1
# permanent
echo "net.ipv4.ip_forward=1" | tee -a /etc/sysctl.conf
```

Vérifier/forcer les passerelles par défaut:
- lan-cli:
```bash
ip route
ip route replace default via 10.10.1.1
```
- dmz-srv:
```bash
ip route
ip route replace default via 10.10.2.1
```

Tests à capturer
- lan-cli → ping dmz-srv:
```bash
ping -c 3 10.10.2.10
```
- dmz-srv → ping lan-cli:
```bash
ping -c 3 10.10.1.10
```
- Table de routage (une machine au moins):
```bash
ip route
```

Diagnostic rapide (si un seul sens fonctionne)
- Souvent la passerelle par défaut manquante du côté qui ne répond pas. Corrigé via `ip route replace default ...`.
- Vérifier `sysctl net.ipv4.ip_forward` sur fw.

5) Étape 4 — Pare-feu simple (autoriser ping + HTTP LAN → DMZ)

Objectif
Mettre nftables sur fw, autoriser seulement ce qui est nécessaire.

Installer/activer nftables:
```bash
apk add nftables
rc-update add nftables boot
```

/etc/nftables.conf (adapte les noms d’interface si besoin):
```bash
#!/usr/sbin/nft -f
flush ruleset

define lan_if = "eth0"
define dmz_if = "eth1"
define lan_net = 10.10.1.0/24
define dmz_net = 10.10.2.0/24

table inet filter {
  chains {
    input {
      type filter hook input priority 0;
      policy drop;

      ct state established,related accept
      iif "lo" accept

      # Ping vers fw depuis LAN/DMZ (optionnel)
      ip saddr { $lan_net, $dmz_net } icmp type { echo-request, echo-reply } accept

      # SSH d’admin depuis LAN vers fw (optionnel)
      iifname $lan_if tcp dport 22 accept

      log prefix "FW_INPUT_DROP " flags all counter drop
    }

    forward {
      type filter hook forward priority 0;
      policy drop;

      ct state established,related accept

      # Autoriser LAN -> DMZ : ICMP (ping)
      iifname $lan_if oifname $dmz_if ip protocol icmp accept

      # Autoriser LAN -> DMZ : HTTP (80)
      iifname $lan_if oifname $dmz_if tcp dport 80 accept

      # Tout le reste bloqué (log rate-limité)
      limit rate 5/second log prefix "FW_FWD_DROP " flags all counter drop
    }

    output {
      type filter hook output priority 0;
      policy accept;
    }
  }
}
```

Appliquer et vérifier:
```bash
nft -f /etc/nftables.conf
rc-service nftables start
nft list ruleset
```

Tests à capturer
- lan-cli → ping dmz-srv: doit PASSer
```bash
ping -c 3 10.10.2.10
```
- lan-cli → HTTP dmz-srv: doit PASSer
```bash
curl -v http://10.10.2.10/
```
- dmz-srv → ping lan-cli: à toi de décider; avec la conf ci-dessus c’est BLOQUÉ (pas de règle DMZ→LAN).
```bash
ping -c 3 10.10.1.10
```
- Logs de fw pour un flux bloqué:
```bash
tail -n 50 /var/log/messages
```

6) Étape 5 — Politique stricte + logs et observation

Objectif
Tout interdit par défaut, autoriser seulement les flux nécessaires, observer autorisé/bloqué.

La config nftables ci-dessus est déjà en “policy drop” (stricte). Il reste à:
- Montrer un paquet autorisé (ex: LAN→DMZ HTTP)
- Montrer un paquet bloqué (ex: LAN→DMZ SSH port 22 vers dmz-srv, ou DMZ→LAN ping)

Générer un flux bloqué depuis lan-cli:
```bash
# Exemple: SSH vers dmz-srv non autorisé
nc -vz 10.10.2.10 22
# ou ping DMZ -> LAN depuis dmz-srv si tu veux montrer ICMP bloqué
```

10) Questions traitées

1. Rôle du pare-feu fw entre LAN et DMZ
- Il segmente et contrôle les flux entre un réseau interne (LAN) et une zone exposée (DMZ), n’autorisant que les services nécessaires et bloquant le reste. Il applique une politique de sécurité et fournit de la visibilité (logs).

2. Pourquoi placer un serveur en DMZ plutôt qu’en LAN interne ?
- La DMZ isole les services exposés (ex: web) du cœur du réseau. En cas de compromission du serveur, l’attaquant ne se retrouve pas directement dans le LAN. Les règles du pare-feu limitent les mouvements latéraux.

3. Différence « tout autoriser sauf… » vs « tout interdire sauf… » — laquelle est la plus sûre ?
- « Tout autoriser sauf… »: on part ouvert, on ajoute des exceptions, risque d’oublier des ports/flux.
- « Tout interdire sauf… »: on part fermé, on n’ouvre que le strict nécessaire; c’est la plus sûre, car elle réduit la surface d’attaque et rend explicites les besoins.

4. Exemple concret de flux bloqué et pourquoi dangereux
- Exemple: TCP 22 (SSH) LAN → dmz-srv bloqué. Laisser SSH ouvert vers un serveur exposé augmente le risque de brute-force et d’exploitation d’une éventuelle faille SSH. On préfère restreindre l’admin via fw ou un saut d’admin, et journaliser.
